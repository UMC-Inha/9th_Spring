### 2️⃣ Swagger와 Rest Docs의 장단점 비교하기

## **Swagger (OpenAPI) 장단점**

### 장점

| 장점 | 설명 |
| --- | --- |
| **자동 문서화** | 코드에 어노테이션만 적으면 즉시 문서 생성됨 |
| **UI 제공** | Swagger UI를 통해 테스트 가능한 인터랙티브한 문서 제공 |
| **개발자 친화적** | 개발 중 API 탐색·테스트가 매우 편함 |
| **빠른 적용 가능** | 의존성 + 간단한 설정만으로 바로 사용 가능 |
| **Client SDK 자동 생성** | OpenAPI 스펙으로 클라이언트 코드 자동 생성 가능 |

### 단점

| 단점 | 설명 |
| --- | --- |
| **문서 정확성 ↓** | 개발자가 어노테이션을 빠뜨리면 문서와 실제 동작이 불일치 하는 경우 많음 |
| **운영 환경에 노출 시 위험** | Swagger UI가 운영에 그대로 열려 있으면 API 공개 위험 |
| **어노테이션 오염** | Controller/DTO에 Swagger 전용 어노테이션이 들어가 코드가 지저분해짐 |
| **테스트 기반이 아님** | 문서가 “실제로 검증된 API”인지 보장이 없음 |

## **REST Docs 장단점**

### 장점

| 장점 | 설명 |
| --- | --- |
| **테스트 기반 문서화 → 정확성 최상** | 테스트가 깨지면 문서 생성이 안됨 → 문서 최신성 자동 보장 |
| **코드 의존도가 낮음** | Swagger처럼 DTO/Controller에 어노테이션이 거의 필요 없음 |
| **운영 환경에 UI가 노출되지 않음** | HTML 문서를 빌드 시점에 생성 → 보안상 안전 |
| **문서의 커스터마이징 자유도 높음** | Asciidoctor 기반으로 매우 깔끔하고 구조적인 문서를 만들 수 있음 |

### 단점

| 단점 | 설명 |
| --- | --- |
| **초기 설정 복잡** | Asciidoctor, Gradle 설정, MockMvc 연동 등 설정량이 많음 |
| **테스트 코드 작성 비용 증가** | API 문서 생성을 위해 상세한 테스트를 작성해야 함 |
| **UI 미제공** | Swagger UI처럼 즉시 테스트 가능한 페이지가 없음 |
| **학습 곡선 있음** | 스니펫 구조, Asciidoc, Test 기반 문서화를 이해해야 함 |
