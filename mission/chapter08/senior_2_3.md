### 3️⃣ 언제 적용하면 좋을 지 파악하기

두 방식은 서로 우열 관계가 아니다. **프로젝트 성격 / 팀 문화 / 인프라 환경**에 따라 선택이 달라진다.

## Swagger가 적합한 경우

### 1. 빠르게 API 문서가 필요할 때

Swagger는 설정이 매우 간단하여, 의존성만 추가하면 즉시 API 문서를 생성할 수 있다.

초기 개발 단계에서 문서화에 시간을 많이 들일 수 없을 때 특히 유용하며, 빠른 시일 내에 문서를 확보해야 하는 프로젝트 상황에 적합하다.

### 2. 프론트/백엔드 협업을 빠르게 시작해야 할 때

Swagger는 UI를 제공하여 API를 직접 눈으로 확인하고 테스트할 수 있다.

프론트엔드 개발자가 API 응답 구조를 바로 확인하고, 실제 요청도 UI에서 즉시 시도해볼 수 있기 때문에

백엔드 개발과 프론트 개발을 병렬로 진행하기에 매우 작은 friction만 존재한다.

### 3. 내부 API 위주로 개발하는 환경일 때

외부 사용자에게 API를 노출할 필요가 없는 환경에서는 Swagger UI가 노출되는 위험이 줄어든다.

오히려 팀원들이 API를 탐색하고 테스트하는 데 큰 편리함을 제공하며,

내부 개발용 API라면 Swagger의 장점이 극대화된다.

### 4. 시니어가 적고 API 설계·문서화 경험이 부족한 팀일 때

Swagger는 문서를 자동으로 생성해 주기 때문에 학습 부담이 적다.

별도의 AsciiDoc 규칙이나 테스트 기반 문서 생성을 이해할 필요 없이,

Controller와 DTO만 존재하면 기본 문서를 바로 생성할 수 있다.

API 변경 시에도 Swagger UI에서 즉시 반영되므로 협업 시 혼선을 줄일 수 있다.


## REST Docs가 적합한 경우

### 1. API 문서의 정확성과 신뢰성이 매우 중요한 프로젝트일 때

REST Docs는 테스트 기반으로 문서를 생성하기 때문에, 실제 API가 동작해야만 문서가 생성된다.

즉, “테스트가 깨지면 문서도 깨진다”라는 구조 덕분에 문서가 항상 최신 API 상태와 일치한다.

외부 고객에게 제공하는 퍼블릭 API나, 문서 정확도가 절대적으로 중요한 환경에서 특히 강력하다.

### 2. 장기·대규모 프로젝트로 문서 유지 비용이 큰 경우

프로젝트가 커질수록 API 문서와 실제 코드가 불일치하는 문제가 자주 발생한다.

REST Docs는 테스트를 통해 문서를 생성하기 때문에, API가 변경되면 테스트도 함께 수정해야 하며

그 과정에서 문서도 자동으로 업데이트된다.

문서 관리 부담이 장기적으로 줄어드는 구조라 대규모 시스템에서 큰 이점을 가진다.

### 3. 문서를 고품질로 구성해야 하고 커스터마이징이 필요한 경우

REST Docs는 Asciidoctor 기반 문서 생성 방식을 사용하기 때문에

문서 구조·서식·디자인을 자유롭게 꾸밀 수 있다.

회사 브랜드 가이드에 맞춘 문서 스타일링이나 복잡한 가이드 문서 작성이 가능해

외부 개발자용 Developer Portal 같은 곳에 사용하기 적합하다.

### 4. 운영 환경에서 API 문서를 노출하고 싶지 않을 때

REST Docs는 빌드 타임에 미리 HTML 문서를 생성하므로

Swagger처럼 운영 서버에서 API 문서를 제공할 필요가 없다.

보안 측면에서 API 문서를 노출하지 않는 것이 중요한 시스템에서 매우 안전한 방식이다.

### 5. 테스트 작성 문화를 강화하고 싶은 팀일 때

REST Docs는 문서를 만들기 위해 테스트가 필수이므로

자연스럽게 **TDD/테스트 중심 개발 문화**가 자리 잡는다.

자동화된 테스트를 활용해 문서의 정확성과 안정성을 동시에 확보할 수 있다.
